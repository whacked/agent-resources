# Bidirectional supersession: add `superseded_by`, materialize the reverse edge, carve out frontmatter mutability

Add a reverse supersession edge to notes and reports. `supersedes:` stays the single-valued, authoritative forward edge on the newer (immutable) file; a new `superseded_by:` list on the older file materializes "who replaced me" so obsolescence is visible when browsing raw files or Obsidian. This is the single consolidated record of the supersession policy, which previously existed only as scattered one-line directives.

# Background

`supersedes:` was a forward-only scalar string in both schemas. Reading notes forward in time, an agent or human could not tell whether a file had been obsoleted without scanning every later file for a `supersedes:` pointing back — forward traversal is a local read, reverse traversal was a full scan. The repo's append-only bias ("never edit a prior file") also meant an obsoleted file carried no local signal of its own obsolescence, even though the reports schema already defined an unreachable `status: "superseded"` value.

Two mechanisms coexisted: the scalar `supersedes:` field (schema-backed, live in reports) and a typed `relations: [{type, target}]` array documented only in the synthesize skill — in no schema and with zero real uses in the vault.

# Analysis

- **Datatype.** `supersedes?: string` — single scalar in both schemas; never an array. No prior design record justified single-cardinality; it was simply the declaration. The immutability rationale lived in the tfq upstream and in terse directives.
- **tfq already answers the reverse query.** As of `20260719.4dd7426`, `tfq --backlinks <old> --relation supersedes` returns every successor (verified, including forks) — so the *tooling* reverse-traversal need is met by the forward edge alone. Materialization is justified specifically by human/Obsidian visibility and O(1) local reads, not by tooling.
- **tfq does not infer the inverse.** Writing `supersedes` does not auto-write `superseded_by`; each direction is written independently. Bidirectional consistency is therefore a policy owned here, not by the binary.
- **`--field-list` replaces.** tfq's list writer replaces the whole list, and the complete successor set is derivable from the forward edges — so the reverse cache is a projection to be *recomputed and replaced*, never blind-appended.
- **CUE schemas are open**, so declaring `superseded_by` is not required for validation to pass; it is declared to pin the `[...string]` type and document intent.

# Decision

1. **Field model.** Add `superseded_by?: [...string]` to `schemas/reports.cue.template.md` and `skills/notes/schemas/notes.cue.template.md`. Keep `supersedes?: string` scalar. Add optional `status?: "current" | "superseded"` to the notes schema (reports already have the enum). Values are bare slugs.
2. **Source of truth.** The forward `supersedes` edge on the immutable successor is authoritative. `superseded_by` is a derived cache: the full set = `tfq --backlinks <ref> --relation supersedes`.
3. **Immutability carve-out.** A prior file's **body** stays immutable. `superseded_by` and `status` are the only frontmatter fields that may change post-hoc, solely to maintain the reverse edge.
4. **Write both ends.** When superseding, set `supersedes` on the new file and recompute+write the old file's `superseded_by` (+ `status: superseded`). Documented in the notes skill, synthesize skill, and `docs/agent-guides/reports.md`.
5. **Doctor janitor.** `skills/doctor/scripts/supersession-repair.sh` recomputes each predecessor's `superseded_by` from the forward edges and, with `--fix`, writes it back (fork-safe, idempotent). `doctor` reports drift; forward wins on any disagreement. Run it on any repo to guarantee both link directions exist.
6. **`relations[]` retired.** The dead typed-relations example is removed from the synthesize skill. Supersession uses the dedicated fields. The sanctioned future form for other typed edges is a map `relations: {<type>: [<slug>…]}`, introduced only when real data appears — not before.

# Consequences

- Reverse traversal is now O(1) for humans (visible in the file) and a single `tfq --backlinks --relation supersedes` for tooling. The obsolescence of a note is legible without running a tool.
- The append-only invariant is relaxed in exactly one bounded way (two frontmatter fields); the audit-trail guarantee on bodies is unchanged.
- Existing reports validate unchanged against the expanded schemas; the new field is optional and type-checked (a scalar `superseded_by` now fails validation).
- Known limitation (deferred): many-to-one *merges* (one file superseding several) remain inexpressible via the scalar `supersedes` — revisit with a scalar→list bump or the relations map.
- `tfq` support is verified present in `20260719.4dd7426`; no binary handoff remains.
- Verified end-to-end: full `scripts/tests/run-tests.sh` green (10/10) including a new `test-supersession.sh` covering schema type-enforcement and the janitor's detect→fix→idempotent cycle.
