# Agent-runner container: Nix owns the build contract, Docker builds the image

Add `agent-runner-container/`, a Dockerized runner for LLM coding agents
(`claude-code`, `codex`, `pi`, `opencode`) where Nix is the single top-level
build orchestrator and Docker performs an intentionally impure image build. The
canonical interface is `nix run .#build-image`; `nix run .#up` goes from a
checkout to a shell in the container in one command.

# Context

We needed a reproducible, easy-to-modify way to run several LLM agents in a
container that streamlines the dev → deploy environment. Constraints:

- Agents should come from `numtide/llm-agents.nix` and be trivial to change.
- It must build on Apple Silicon macOS with Docker Desktop, where the host has
  **no Nix Linux builder** (`extra-platforms` lists only `x86_64-darwin`; no
  `*-linux` system features). Cross-compilation is explicitly *not* a goal.
- The container's main user is `agent`; the host user's uid/gid map onto `agent`
  so bind-mounted files keep host ownership.
- `host.docker.internal` must be reachable; `$PWD` binds to `/workspace`; root
  access is via `docker exec` after start.

A governing directive scoped the build approach: Nix defines the dev shell and
exposes the canonical image build command; Docker executes the build and may do
lockfile-controlled networked installs; do not force pure-Nix packaging, a
NixOS base, `dockerTools`, or a remote Linux builder for v1.

# Reasoning

A pure `dockerTools.buildLayeredImage` would need a Nix Linux builder on this
Mac — friction the directive rules out. A plain Debian + `npm` image would honour
the directive's impurity model but drop `llm-agents.nix` and fail for non-npm
agents (e.g. `pi`). The reconciliation: build `FROM nixos/nix` and install the
agents *inside* the Docker build via `nix profile install` from the pinned
`llm-agents` flake. Nix runs in Docker's native Linux VM, so no host Linux
builder is needed; binaries come from `cache.numtide.com` (root in the build is
trusted), so nothing compiles; and `flake.lock` provides the lockfile control the
directive requires.

Keeping Nix in the runtime image is a deliberate v1 trade-off: it is heavier
(~4.5 GB) but lets a user `nix profile install` more agents at runtime, which
suits a dev-runner. Slimming via `nix2container`/multi-stage is deferred. The
impurity requirement is satisfied concretely by pulling a helper script from
GitHub at build time, pinned by commit and verified by `sha256` — impure, but
frozen, so the build fails if upstream content drifts.

The `nixos/nix` base required three non-obvious fixes, all verified end-to-end:
TLS had to point at our own GC-rooted `cacert` (the base bundle is dropped by
`nix-collect-garbage`); `/etc/{passwd,group,shadow}` are read-only store symlinks
and must be materialised before `shadow` can add the `agent` user; and login
shells read `.bash_profile`, so the impure script is loaded via
`.bash_profile → .bashrc → /etc/profile.d`.

# Decision

- New self-contained directory `agent-runner-container/` containing
  `flake.nix`, `flake.lock`, `Dockerfile`, `entrypoint.sh`, `run.sh`,
  `Makefile`, `.dockerignore`, and `README.md`.
- `flake.nix` exposes `devShells.default`, `apps.{build-image,run,up,root,add,
  test-image}` (default = `up`), and `packages.{agent-env,container-tools}`. The
  agent lineup is one editable `agentNames` list — the single source of truth.
- The image is `FROM nixos/nix` pinned by digest; agents + plumbing install into
  a shared GC-rooted profile; an impure pinned+hashed GitHub pull demonstrates
  the hybrid contract; plain `nix-collect-garbage` trims build-only paths last
  (NOT `-d`: deleting old generations orphans the base image's default profile
  and removes `nix` itself, breaking runtime package installation).
- The runtime is a plain Docker image; the `run`/`up` apps and `run.sh` are thin
  wrappers over `docker run`. Nix stays in the image, so packages can be added at
  runtime — declaratively (edit the flake, rebuild) or imperatively and
  ephemerally via `nix run .#add` (`docker exec -u 0 … nix profile install` into
  the shared profile, landing on the agent's PATH). `run.sh` accepts extra
  `docker run` flags via `DOCKER_ARGS` for additional mounts/ports/env.
- `entrypoint.sh` (PID 1, root) remaps `agent` to `HOST_UID`/`HOST_GID`, then
  drops to `agent` via `gosu`. No `USER` is set, so `docker exec` defaults to
  root for on-demand privilege.
- `run.sh` is the canonical launcher: binds `$PWD` (or `$WORKSPACE`) to
  `/workspace`, adds `host.docker.internal:host-gateway`, and passes the host
  uid/gid. `nix run .#up` = build (cached) + prune dangling + attach.

# Consequences

- `nix run .#build-image` and `nix develop` both work on aarch64-darwin with no
  Linux builder; the same entry point is usable in CI.
- Verified end-to-end: all four agents on `PATH`; the pinned impure script
  present and hash-matched; `agent` remapped to the host uid (a file written in
  `/workspace` is owned by the host user); `host.docker.internal` resolves and is
  live-reachable; `docker exec` defaults to root, `-u agent` gives `agent`;
  `nix run .#add -- hello` installs into the running container and `hello` appears
  on the agent's PATH; `DOCKER_ARGS` injects extra mounts correctly.
- Runtime package installation works because `nix` survives the (non-`-d`)
  garbage-collect: `agent` can run `nix` for inspection (world-readable default
  profile), and root (via `docker exec`) can write the store. Imperative installs
  are ephemeral under `--rm`; declarative rebuilds persist.
- Reproducibility rests on `flake.lock`, the digest-pinned base image, and the
  commit+`sha256`-pinned GitHub pull (identically-working, not byte-identical).
- Known limitations, documented in the README: ~4.5 GB image (slimming deferred
  to a future `nix2container`/multi-stage step); on macOS Docker Desktop the
  bind-mount *gid* may display differently than inside the container (uid maps
  correctly; Linux maps both); a fast host `nix develop` needs `cache.numtide.com`
  trusted in the host's Nix settings.
- Changing the agent set is a one-line edit to `agentNames` plus a rebuild.
