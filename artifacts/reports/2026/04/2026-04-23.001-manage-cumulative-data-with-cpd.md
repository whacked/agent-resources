# Manage Cumulative Data with CPD

Require agents to use schema-bearing CPD YAML for accumulated record data instead of ad hoc JSONL streams.

# Context

Extended exploratory analysis, data generation, and ETL work often produces many JSON-like records over time. JSONL is easy to append, but it does not carry an enforced schema or a compact provenance-friendly structure. As tasks evolve, record shape drift becomes hard to audit without reconstructing intent from scripts and historical output.

The local `cpd` binary supports CPD YAML with `_schemas`, `_columns`, join tables, a configurable `data` section key, and a configurable `...` catch-all column. Its examples show that `_schemas.data...` describes one expanded record, while the `data` table may store compact positional rows keyed by `_columns`. Plain positional columns preserve structured object and array values; only catch-all columns flatten object fields into the expanded record.

# Analysis

Agents usually know the output shape while writing the scripts that generate records. Capturing that shape before accumulation starts makes the data artifact easier to validate, migrate, and resume. Using the default `data` table and `...` catch-all column keeps CPD files predictable, while allowing explicit overrides preserves compatibility with CPD's documented flags.

Schema changes need different handling from ordinary appends. A pure append under the same schema does not change the system design. Schema expansion, migration, replacement, or incompatible new files do change the data contract and should be recorded as normative reports under the existing directive. Date shards organize artifacts by task, workflow, or batch; they are not automatic daily rotation boundaries.

# Decision

Add an `AGENTS.md` directive requiring cumulative structured records to be stored as CPD YAML under date-sharded artifact paths. The directive tells agents to decide the expanded record shape before accumulation, encode it under `_schemas.data...` with CUE, and store compact rows with deliberately chosen `_columns`, join tables, data columns, and the default `...` catch-all. It also preserves CPD's default names unless an explicit `-data-key` or `-payload-column` override is needed.

The directive also defines continuation behavior: append unchanged records under the existing schema and accumulation scope, update compatible schemas before appending changed records, migrate compatible-but-normalized shapes into a new CPD file, and start a separate CPD file for incompatible shapes.

# Results

- Agents get a standard location for cumulative CPD files: `artifacts/records/YYYY/MM/DD/<prefix><slug>.cpd.yaml`, where the date shard is chosen for the task, workflow, or batch.
- CPD files carry their intended data contract in `_schemas` instead of relying on retrospective JSONL analysis.
- New CPD files follow current `cpd` defaults, including `...` as the catch-all column.
- Schema changes and migrations are covered by the existing normative report requirement.
- Pure appends remain lightweight and do not require new reports.
