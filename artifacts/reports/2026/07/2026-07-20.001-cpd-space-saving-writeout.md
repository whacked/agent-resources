# CPD write-time space-saving directives

Add three normative write-time compaction directives to the CPD agent guide: null as `~`, proactive join tables for known categoricals, and pushing a size-dominating payload to a trailing block-scalar column. Prior to this the guide covered schema shaping, storage, and lifecycle but gave no guidance on keeping accumulated files small or readable.

# Motivation

CPD files accumulate append-only across many sessions, so a per-row byte cost compounds over the life of a dataset, and a compaction choice made at write time cannot be revised later without a migration. Agents writing CPD had no directive on compaction, so they defaulted to whatever `cpd` emits or to verbose hand-authoring. Three patterns recur and are cheap to apply at write time yet expensive to retrofit.

# Analysis

Each directive was verified against the installed `cpd` (`devsync/yayamdb/cpd`) by round-tripping YAML → JSONL:

- **Null as `~`.** `~` is valid YAML null in positional slots and inside splat dicts; `cpd` already emits `~` when it compacts join-column nulls. Saves 3 bytes per occurrence (`null` → `~`).
- **Join tables for known categoricals.** `cpd` auto-promotes low-cardinality strings, but auto-detection only sees the current batch. When the agent knows a field is categorical up front (campaign, publisher, book title, artist, album, model id), forcing the join table (`-join-tables`, or a hand-authored `_columns` entry + `name: int` map) stores the repeated string once and lets each row carry a small integer.
- **Size-dominating payload last, as a block scalar.** A YAML block scalar (`|-`) cannot appear inside a flow `[...]` row, so the row must be written in block style. Confirmed both forms round-trip: a dedicated trailing named column holds the bare `|-` block; the splat `...` catch-all needs a dict wrapper (`{lyrics: |- ...}`) so the field key survives expansion. This trades row height for human readability and is warranted only when a payload actually dominates row size — for uniformly structured rows, compact flow arrays with the `~` trick remain smaller and more scannable.

These are advisory heuristics applied during writeout; they change neither the CPD format nor any existing dataset's schema, so no existing data file requires migration.

# Decision

`docs/agent-guides/cpd-data.md` gains a `## Write compactly` section stating the three directives with a worked block-scalar example, placed between record shaping and storage. The existing shaping example is updated to use `~` for null, dogfooding directive 1.

# Consequences

- Agents writing or appending CPD are now directed to compact at write time; the three techniques are normative guidance, not format changes.
- No schema change, no data migration, no data-file replacement — existing CPD files remain valid and are not rewritten.
- Directive 3 is explicitly conditional ("only when a payload actually dominates"); the default for structured data stays compact flow arrays.
- This directive change is recorded here per the CPD guide's own closing rule and the repo supersession/report invariants.
